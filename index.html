01
Flagship backend system Live API

Smart Job Scheduler

A database-backed scheduling service designed around a hard distributed-systems problem: ensuring competing application instances cannot execute the same claim while still recovering work abandoned by a failed worker.

REST API→Priority policy→Locked claim→Handler registry→State commit
Architecture decision

Short REQUIRES_NEW transactions lock and claim rows; pluggable handlers execute after the transaction closes; completion reacquires the lock and verifies ownership.

Reliability model

Optimistic versioning, pessimistic claim locks, expiring leases, opaque claim tokens, bounded retries, and exponential backoff protect each lifecycle transition.

Trade-off

The safety-first claim query can make competing dispatchers wait briefly on locked rows. It favors correctness and portability; PostgreSQL SKIP LOCKED is the next throughput optimization.

Verification & delivery

18 passing tests cover concurrency and lifecycle invariants. Flyway migrations, environment-specific profiles, a multi-stage Docker image, and PostgreSQL configuration make releases reproducible.

02
Explainable text analysis Live demo

Sentiment Analysis Service

A deterministic lexical analysis service that deliberately prioritizes explainability, predictable latency, and zero external inference cost over the broader language understanding of a trained model.

Thymeleaf / REST→Tokenizer→Lexicon→Signal scorer→Explanation
Architecture decision

Tokenization, lexicon ownership, contextual scoring, and explanation are separate components behind one application service, keeping both the server-rendered UI and versioned REST API consistent.

Language handling

Code-point-safe emoji matching and boundary-aware word detection prevent surrogate-pair corruption and substring false positives; negators and amplifiers adjust nearby signals.

Trade-off

Lexical rules are fast, inspectable, and locally executable, but cannot fully model sarcasm or domain-specific nuance. Explanations make that limitation visible rather than hiding uncertainty.

Verification & API

21 passing unit and integration tests protect scoring, contextual safety signals, and HTTP contracts. Structured DTOs, validation errors, OpenAPI documentation, and Docker packaging provide a production-shaped boundary.
